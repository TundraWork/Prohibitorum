# Simplified Chinese (zh) Localization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers-extended-cc:subagent-driven-development (recommended) or superpowers-extended-cc:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a complete, high-quality Simplified Chinese (`zh`) locale to the dashboard SPA, derived from a lightly-normalized English source, with persistence + browser detection and a prod-compiler/key-parity guard covering both locales.

**Architecture:** `zh.ts` mirrors `en.ts` key-for-key. It is first scaffolded as a copy of the normalized English (so key-parity + compile guards pass immediately), then translated namespace-by-namespace against a fixed glossary, then consistency-swept. A `useLocale` composable (mirroring the existing `useTheme`) persists the choice to `localStorage` and detects `navigator.language` on first load. The `LocaleSwitcher` is already locale-agnostic and needs no change.

**Tech Stack:** Vue 3, vue-i18n (Composition mode, `legacy: false`), `@intlify/message-compiler` (prod-parity guard), vitest, vue-tsc; Go `go:embed` for the built SPA.

**Spec:** `docs/superpowers/specs/2026-06-19-cn-i18n-design.md`

## Translation Style Guide & Glossary (the contract for every translation task)

**Register:** `你`, pronoun-light. Omit "you/your" where Chinese reads naturally (`管理账户`, not `管理你的账户`); use `你` (never `您`) only when a pronoun is genuinely needed.

**Rules:**
- Buttons/labels terse; descriptions are full sentences ending in `。`. Use full-width punctuation (`，。、：？！`) in prose; ASCII inside code/identifiers and interpolation.
- **Preserve every interpolation param verbatim:** `{count}` `{name}` `{client}` `{displayName}` `{date}` `{when}` `{app}` `{idp}` `{seconds}` `{n}` `{username}`.
- **Preserve verbatim:** CLI commands (`prohibitorum enroll-admin`), URLs, code identifiers, and any existing vue-i18n escapes — notably `{'@'}` (literal @) and the brace-escapes `{'{'}` / `{'}'}` in `admin.saml.attributeMapHint`.
- **No literal `@`** unescaped (escape as `{'@'}`), **no literal `|`** (plural separator), no unescaped `{`/`}` in any Chinese string.
- A space between CJK and a Latin token is fine (`OIDC 应用`, `5 MB`).

**Glossary A — concepts → Chinese (technical lean):** passkey 通行密钥 · sign in/out 登录/退出登录 · session 会话 (active 活动会话, this device 当前设备) · password 密码 · recovery code 恢复码 · authenticator app 身份验证器应用 (short 身份验证器) · one-time/TOTP code 一次性验证码 · consent/approve access 授权/授权访问 · permission (user-facing scope) 权限 · enrollment/set up account 注册/设置账户 (enrollment link 注册链接) · invitation 邀请 · connected accounts 已关联账户 (connect/disconnect 关联/解除关联) · identity provider 身份提供商 · display name 显示名称 · username 用户名 · account 账户 · device/pairing/pairing code 设备/配对/配对码 · avatar 头像 · group 用户组 · admin/user (roles) 管理员/用户 · audit log 审计日志 · signing key 签名密钥 · Danger zone 危险操作 · Enable/Disable 启用/停用.

**Glossary B — keep English:** `OIDC` `SAML` `TOTP` `WebAuthn` `JWK`/`JWKS` `PKCE` `ACS` `NameID` `EntityID` `IdP`, Client ID / `client_id`, redirect URI, `HTTP-POST`/`HTTP-Redirect`, metadata XML, and **scope values verbatim** (`openid` `profile` `email` `offline_access` `groups` `address` `phone`). Admin protocol field labels lean bilingual/English (e.g. `Scopes（授权范围）`; keep `Redirect URIs`, `Entity ID`, `NameID format`).

**Glossary C — common verbs:** 保存 / 取消 / 继续 / 关闭 / 添加 / 删除 / 移除 / 复制 / 已复制 / 验证 / 确认 / 允许 / 拒绝 / 启用 / 停用 / 生成 / 激活 / 轮换 / 退役.

/**
 * Prod-parity i18n guard, all locales.
 *
 * vue-i18n's DEV compiler only WARNS on malformed messages (e.g. a literal '@',
 * read as a linked reference); the PRODUCTION compiler THROWS at render time,
 * blanking the subtree. This compiles every leaf of every locale through
 * @intlify/message-compiler with a throwing collector — the prod path — so a
 * compile-invalid message fails here, not in a user's browser. Literal '@' must
 * be escaped as {'@'}; '{...}' interpolation and '|' plurals stay valid.
 */
import { describe, it, expect } from 'vitest'
import { baseCompile } from '@intlify/message-compiler'
import en from './en'
import zh from './zh'

function leaves(obj: unknown, path: string[] = []): Array<[string, string]> {
  if (typeof obj === 'string') return [[path.join('.'), obj]]
  if (obj && typeof obj === 'object') {
    return Object.entries(obj as Record<string, unknown>).flatMap(([k, v]) => leaves(v, [...path, k]))
  }
  return []
}

describe.each([['en', en], ['zh', zh]])('%s locale messages compile under the production compiler', (name, msgs) => {
  it(`has no message that the prod vue-i18n compiler rejects (${name})`, () => {
    const broken: string[] = []
    for (const [key, value] of leaves(msgs)) {
      const errors: string[] = []
      baseCompile(value, { onError: (e) => errors.push(`${e.code}:${e.message}`) })
      if (errors.length) broken.push(`${key} = ${JSON.stringify(value)} -> ${errors.join('; ')}`)
    }
    expect(broken, `messages rejected by the production compiler:\n${broken.join('\n')}`).toEqual([])
  })
})
```
Then delete the old test: `git rm dashboard/src/locales/en.compile.test.ts` (or `rm` + stage the deletion).

- [ ] **Step 4: Create the key-parity test.**

Create `dashboard/src/locales/locales.parity.test.ts`:

/**
 * Key parity: zh must define exactly the same keys as en — no missing keys
 * (which would silently fall back to English) and no extra keys (dead strings).
 * This is the guard against translation drift as en evolves.
 */
import { describe, it, expect } from 'vitest'
import en from './en'
import zh from './zh'

function keys(obj: unknown, path: string[] = []): string[] {
  if (typeof obj === 'string') return [path.join('.')]
  if (obj && typeof obj === 'object') {
    return Object.entries(obj as Record<string, unknown>).flatMap(([k, v]) => keys(v, [...path, k]))
  }
  return []
}

describe('locale key parity', () => {
  it('zh defines exactly the same keys as en', () => {
    const enKeys = new Set(keys(en))
    const zhKeys = new Set(keys(zh))
    const missing = [...enKeys].filter((k) => !zhKeys.has(k))
    const extra = [...zhKeys].filter((k) => !enKeys.has(k))
    expect({ missing, extra }).toEqual({ missing: [], extra: [] })
  })
})
```

```ts
import { watch } from 'vue'
import { i18n } from '@/i18n'

const STORAGE_KEY = 'locale'

/**
 * Single source of truth for the app locale. Persists the user's choice to
 * localStorage['locale']; on first load with no stored choice, detects the
 * browser language (zh* -> 'zh', else 'en'). Call once at app start (App.vue)
 * so it applies on every route, including pre-login pages. LocaleSwitcher binds
 * the global locale directly; the watcher here persists whatever it sets.
 *
 * CSP is `script-src 'self'`, so the locale is applied during Vue bootstrap
 * (no inline pre-paint script); text reflow is instant.
 */
function resolveInitial(): string {
  const available = i18n.global.availableLocales as string[]
  const stored = localStorage.getItem(STORAGE_KEY)
  if (stored && available.includes(stored)) return stored
  const nav = (navigator.language ?? '').toLowerCase()
  if (nav.startsWith('zh')) return 'zh'
  return 'en'
}

export function useLocale() {
  const locale = i18n.global.locale // WritableComputedRef<string> (legacy: false)
  const initial = resolveInitial()
  if (locale.value !== initial) locale.value = initial
  watch(locale, (v) => {
    try { localStorage.setItem(STORAGE_KEY, v) } catch { /* storage unavailable */ }
  })
  function setLocale(l: string): void { locale.value = l }
  return { locale, setLocale }
}
```

## Review follow-ups (in-cycle, non-blocking)

- **F-1 (Task 3 code review, Minor, test-only):** `useLocale.ts` uses `watch(locale, …, { flush: 'sync' })`, which deviates from the plan template and the `useTheme` pattern (no explicit flush). It's harmless in production (`useLocale()` is called once in `App.vue`) but in tests each `useLocale()` call registers another sync watcher on the module-singleton `i18n.global.locale` with no `afterEach` cleanup, and the persistence test's `await nextTick()` is non-load-bearing under `sync`. Resolve opportunistically by dropping `flush: 'sync'` (default `'pre'` aligns with `useTheme`; the test's `nextTick` then becomes accurate) and/or reverting `beforeEach` to the plan's `clear()`-then-set order. Not addressed now to avoid re-tuning an empirically-green test mid-cycle.
- **F-2 (final review, Minor, test-robustness):** `locales.params.test.ts`'s token extractor `/\{[^{}]*\}/g` mis-tokenizes the nested-brace escapes in `admin.saml.attributeMapHint` (`{'{'}` / `{'}'}` both extract as `{'}`). Benign today — en and zh share an identical escape sequence so the per-leaf equality still holds, and the prod compile guard (`baseCompile`) validates the escapes render correctly. Worth a clarifying comment (or a smarter extractor) only if that guard is revisited.
