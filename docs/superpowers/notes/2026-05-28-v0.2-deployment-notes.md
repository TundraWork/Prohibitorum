# v0.2 deployment notes — security posture and known gaps

These are operator-facing notes documenting (a) caveats in v0.2's
implementation that don't show up in code reviews but matter for production
deployment, and (b) NIST/OWASP gaps deliberately deferred past v0.2.

Anchored to the master branch as of 2026-05-28, after the three-bundle
audit-fix sequence. Each section names the actual file/line where the
behaviour lives so future-you can verify nothing has drifted.

## 1. Rate limiter is per-process

The HTTP rate limiter at `pkg/authn/ratelimit.go` is an in-memory
fixed-window bucket store. It guards `/auth/password/begin` (30/min/IP),
`/auth/totp/verify` (30/min/IP), `/auth/recovery-code/verify` (30/min/IP),
`/me/sudo/complete` (10/min/session), `/api/prohibitorum/pair/begin`
(10/min/IP), enrollment register (20/min/IP), and a few others — see
`grep -rn 'rateLimit(' pkg/server` for the live list.

**Multi-replica caveat.** A horizontally scaled deployment (N replicas
behind a round-robin LB) multiplies the attacker's budget by N. With
N=4 and the login bucket of 30/min/IP, an attacker can sustain 120
login attempts per minute per source IP before tripping the limiter.

Mitigation options, ordered by simplicity:

1. **Session affinity at the LB.** Pin per-source-IP traffic to a single
   replica. Works for IP-keyed buckets only; account-keyed buckets
   (sudo, pair_lookup, pair_approve) still split across replicas.
2. **Run N=1.** Acceptable for low-volume self-hosted deployments
   where the IdP is not the bottleneck.
3. **Front with an external rate-limiter.** A WAF, API gateway, or
   reverse-proxy with per-IP rate limiting (e.g. nginx
   `limit_req_zone`) gives a deployment-wide bucket independent of the
   replica count.

**Future work.** `pkg/authn/ratelimit.go` is a concrete struct, not an
interface. A Redis-backed implementation would need a small refactor to
extract the `Allow`/`RetryAfter` surface into an interface. Not in v0.2
scope; the persistent `auth_throttle` table already gives durable
per-account-per-factor backoff that survives any HTTP rate-limit
bypass, so this is a defense-in-depth gap rather than a primary line.

## 2. AES-GCM DEK rotation budget

TOTP secrets at rest use AES-256-GCM with 96-bit random nonces. Per
NIST SP 800-38D §8.3 (and Neil Madden's "Stick a fork in it"
analysis), a single 256-bit key can safely encrypt ~2^32 messages
under random 96-bit nonces before the birthday-bound collision risk
approaches 2^-32.

**For TOTP.** One ciphertext per account per (re-)enrollment. A
deployment with 10 million accounts, where each user re-enrolls TOTP
once per year, sits at 10M nonces/year — eight orders of magnitude
below the safe bound. The schema supports rotation via
`totp_credential.key_version`; the AAD (`'totp:' || account_id ||
':' || key_version`) is bound to the row identity so a key roll
doesn't authenticate stale ciphertext from another version.

**Operational guidance.** No action required for any realistic
deployment. A deployment re-enrolling millions of TOTPs per day for
years should plan periodic DEK rotation — generate a new
`DataEncryptionKeys[N+1]`, set it as current, run an automated sweep
that decrypts each `totp_credential` row at its existing `key_version`
and re-encrypts at the new one. v0.7+ task (no schema work needed,
just a one-shot CLI).

## 3. OIDC `auth_time` semantics on sudo

v0.2 stamps `session.SudoUntil` (the Go-side session-data field at
`pkg/authn/middleware.go:27`) on every successful `/me/sudo/complete`.
It does **NOT** update the PG `session.auth_time` column (set once on
session creation at `pkg/session/session.go`).

v0.4 will light up the OIDC token endpoint, at which point `auth_time`
will be surfaced to RPs per OIDC Core 1.0 §2. A decision is needed
BEFORE that endpoint goes live:

- **Treat sudo as a fresh authentication event** → re-stamp
  `session.auth_time` on sudo success. `prompt=login` semantics: any
  RP that requests reauth gets the most recent sudo timestamp.
- **Treat sudo as a within-session elevation** → leave `auth_time`
  pinned to session creation; expose `sudo_until` as a separate
  extension claim or omit it from tokens entirely.

**Spec hook.** NIST SP 800-63B-4 §5.2 is explicit that AAL3 reauth
MUST use the same factors as the original authentication. If sudo
via `password_totp` re-stamps `auth_time`, an RP that expected
AAL2/AAL3 from the original login could mistakenly extend that
assurance to the sudo-elevated session — particularly bad when the
original auth was WebAuthn-only (AAL3-capable) and the sudo was
password+TOTP (AAL2). The safer default is option 2: don't move
`auth_time`. Option 1 needs an `amr` re-write to be sound.

This decision needs to be locked before any v0.4 work that emits
ID tokens.

## 4. Password breach-list check is deferred

`/me/password/set` (handler at `pkg/server/handle_me_password.go`)
currently validates length only. NIST SP 800-63B-4 §5.1.1.2 phrases
the breach-list check as a SHALL:

> Verifiers SHALL compare prospective secrets against a list that
> contains values known to be commonly-used, expected, or compromised.

v0.2 ships without this check; it is a known NIST-SHALL gap.

**Viable approaches** when this lands:

- **HIBP Pwned Passwords k-anonymity API.** Client sends the first
  5 hex chars of SHA-1(password); server returns all suffix matches.
  Plaintext never leaves the IdP. Adds an external network
  dependency; failure mode (HIBP down) must be defined — fail-open
  (allow) or fail-closed (reject). NIST §5.1.1.2 implies fail-open
  is acceptable since the check is informational.
- **Static blocklist.** Embed top ~10k common passwords as a Go
  string slice. No network dep; staler. Reasonable starting point;
  HIBP can be layered later.

Recommendation: ship static blocklist first (a few hundred KB of
constants), gate HIBP behind a config flag for deployments that
want it.

## 5. `auth_throttle` shared across login + sudo surfaces

The `auth_throttle` table is keyed by `(account_id, factor)` where
factor is one of `password`, `totp`, `recovery_code`. Both login
(`/auth/password/begin`, `/auth/totp/verify`,
`/auth/recovery-code/verify`) and sudo
(`/me/sudo/complete` with method `password_totp` or `recovery_code`)
RegisterFailure against the same row.

**Failure mode this exposes.** An attacker with a valid session for
account A who repeatedly hits `/me/sudo/complete` with wrong codes
will lock A out of the *login* path as well — the throttle row is
shared. Symmetrically, repeated failed logins lock the legitimate
user out of sudo on their existing session.

**This is intentional defense-in-depth.** One budget per factor
across all surfaces means an attacker can't multiply their attempts
by enumerating endpoints. The lockout schedule is `[0, 0, 1s, 5s,
30s, 5m, 15m]` (configx default), so even a hostile internal user
can't lock an account for arbitrary periods — the worst sustained
state is 15m per factor.

If a deployment needs to dissociate login throttling from sudo
throttling, the cleanest split would be a separate row keyed by
`(account_id, factor, surface)` where surface ∈ {`login`,
`sudo`}. Not planned for any release; document the current behaviour
so operators understand the cross-surface coupling.

## Audit trail change in this bundle

The throttle now emits `credential_event { factor=<f>, event="factor_locked",
detail={failed_attempts, until} }` on each transition from unlocked → locked.
SOC pipelines can index on `event='factor_locked'` rather than counting
fail rows; existing alerts on `event='fail'` continue to fire as before
(no behaviour change there). The smoke step 45 lower-bound assertions
keep passing because they only check the v0.2-required event types are
present.
