# v0.2 — Password + TOTP design

**Status:** approved 2026-05-25, ready for implementation.

This document is a **delta on top of** the v0.1 multi-protocol rescope spec
(`docs/superpowers/specs/2026-05-24-multi-protocol-rescope-design.md`).
That spec defines the schema (migrations 001–005), package layout, crypto
policies, threat model, and roadmap. Read it first. This delta resolves the
implementation ambiguities the master spec explicitly left open for v0.2 and
locks in concrete decisions for things the master spec only sketched.

## Scope

v0.2 implements the **Password + TOTP** upstream auth method on the v0.1
skeleton. Concretely:

1. Argon2id PHC-string password storage with per-params re-hash on verify.
2. RFC 6238 TOTP enrollment + verify, with AES-256-GCM at-rest secrets,
   ±1 step drift, `last_step` replay prevention.
3. Single-use recovery codes (10 per account, argon2id-hashed) minted at
   TOTP confirmation and regenerable later.
4. The two-step login flow (`/auth/password/begin` → partial session →
   `/auth/totp/verify` | `/auth/recovery-code/verify`).
5. `auth_throttle` populated with **exponential backoff** on every verify
   failure across all three factors; reset on success.
6. Sudo extended to accept WebAuthn **or** password+TOTP **or**
   recovery-code as the elevation factor, so accounts without a passkey
   can still step up before sensitive operations.
7. The WebAuthn-preferred factor policy as an explicit endpoint
   (`POST /me/auth/revoke-password-totp`) callable at any time.
8. Audit events for every register / use / fail / revoke across all three
   factors, plus sudo grants.
9. `cmd/smoke` extended with ~10–12 steps covering the full v0.2 surface.

## Non-goals

- No new migrations. Migration 003 (`password_credential`,
  `totp_credential`, `recovery_code`) and migration 001 (`auth_throttle`,
  `credential_event`) already shipped in v0.1; v0.2 just fills in the
  package code that consumes them.
- No frontend work (`dashboard/` still empty; v0.6).
- No federation / OIDC OP / SAML IdP (v0.3+).
- No TOTP secret re-encryption job on DEK rotation. The `key_version`
  column supports rotation by reading old rows under the old version and
  re-encrypting on next touch; an explicit batch rotation tool is v0.7+.
- No `GET /auth/methods?username=...` discovery endpoint. Account-existence
  disclosure is privacy-sensitive; defer the decision until v0.6 frontend
  pins down the UX. v0.2's login UI (when it lands) shows all methods, and
  the server resolves which is usable per request type.

## Decisions resolved by this delta

### D1. Partial-session-token has no IP/UA binding

The token returned by `/auth/password/begin` and consumed by
`/auth/totp/verify` or `/auth/recovery-code/verify` is bound only to
account identity, not network identity.

**KV value:** `{account_id, factor_completed: "password", issued_at}`
**Key:** `partial_session:<random-43-char-base64url>` (same generator as
session tokens — `crypto/rand` 32 bytes, base64url no padding)
**TTL:** 5 minutes (configurable via `Auth.PartialSessionTTL`)
**Lifecycle:** Single-use. Deleted on first consumption — success **or**
failure. A replay attempt returns 401.

**Rationale:** OWASP MFA / Authentication / WSTG cheat sheets treat IP
and User-Agent as *risk signals*, not as authoritative bindings for
intermediate tokens. Hard-binding to IP breaks legitimate users on
carrier-NAT mobile/WiFi handoff during the 5-minute window with no
security gain proportional to that friction. The defenses that matter
are: short TTL, single-use, cryptographic randomness, per-account
rate-limiting on the step-2 endpoint.

References: OWASP Multifactor Authentication Cheat Sheet; OWASP
Authentication Cheat Sheet; OWASP WSTG-ATHN-11.

### D2. Throttle uses exponential backoff with typo grace

Per `(account_id, factor)` row in `auth_throttle`. Schedule indexed by
`failed_attempts - 1` (0-based), clamped to last index:

```
schedule := []time.Duration{
    0, 0,                              // failures 1 and 2 — typo grace
    1*time.Second, 2*time.Second,      // failures 3, 4
    4*time.Second, 8*time.Second,      // failures 5, 6
    16*time.Second, 32*time.Second,    // failures 7, 8
    1*time.Minute, 2*time.Minute,      // failures 9, 10
    4*time.Minute, 8*time.Minute,      // failures 11, 12
    15*time.Minute,                    // failure 13 and beyond (clamp)
}
lockout := schedule[min(failed_attempts-1, len(schedule)-1)]
```

The first two failures get no lockout (typo grace). From the third
failure on, each adds an exponentially longer lockout, capped at
15 minutes. Once at the cap, additional failures don't extend further —
the row stays at 15-min cycles until a success clears it.

**On verify failure:**

```sql
UPSERT auth_throttle (account_id, factor)
SET failed_attempts = failed_attempts + 1,
    window_start    = COALESCE(window_start, now()),
    locked_until    = now() + schedule[min(failed_attempts+1, len)]
```

**On verify success:**

```sql
DELETE FROM auth_throttle WHERE account_id = $1 AND factor = $2
```

**On verify entry:** If `locked_until > now`, return `429 Too Many
Requests` with `Retry-After: <seconds>` and do **not** perform the
expensive crypto check. (This both protects the CPU and gives the
attacker no oracle for "is this account currently locked?")

**Schedule lives in `configx.Auth.ThrottleSchedule`** as
`[]time.Duration`, with the default above. Operators can tighten (e.g.
to `[1s, 2s, ...]` removing the grace) without code change.

References: OWASP Authentication Cheat Sheet (exponential backoff
endorsed as alternative to fixed lockout); RFC 4226 §7.3 (increasing
delays); OWASP Password Storage Cheat Sheet.

### D3. Username-enumeration defense via constant-time dummy verify

`/auth/password/begin` MUST execute argon2id verification regardless of
whether the username exists or has a password set, so step-1 timing
does not leak account existence.

**Implementation:** package-level constant `dummyArgon2idHash` (a fixed
PHC string hashed from a known dummy password using
`configx.PasswordHashParams`, computed once at package init). When the
account lookup fails or there is no `password_credential` row, run
`argon2id.Verify(submittedPassword, dummyArgon2idHash)` and discard the
result. Return the same 401 body in all failure cases:
`{"code":"bad_credentials","message":"invalid username or password"}`.

**Audit:** On any failure path, emit `credential_event{factor:'password',
event:'fail'}` with `account_id` populated when the lookup succeeded
(so an attacker spraying usernames produces a thin audit trail; legit
typos produce one with `account_id` set).

**Throttle on success of step 1, failure later:** if step 1 (password)
succeeds and a partial session is issued, the password throttle row is
reset. Step 2 (TOTP / recovery code) has its own per-factor throttle.

### D4. TOTP enrollment ergonomics

**otpauth URI format** (per Google's Key URI Format spec; both label
prefix and `issuer` query param set for backward compatibility with
older Google Authenticator versions):

```
otpauth://totp/{Issuer}:{username}?secret={B32}&issuer={Issuer}&algorithm=SHA1&digits=6&period=30
```

`{Issuer}` is `configx.TOTP.Issuer`, a new field defaulting to
`configx.WebAuthn.RPDisplayName` (which itself defaults to
`"Prohibitorum"`). A deployment that brands itself "Acme SSO" sets
`WebAuthn.RPDisplayName = "Acme SSO"` and both passkey and TOTP UX use
the same name without further config.

`{username}` is the account's `username` column, URL-encoded per
RFC 3986. Email-style usernames work fine because `@` is reserved-but-
safe in the path segment after percent-encoding.

**Enrollment flow:**

```
POST /me/totp/begin  (sudo required iff a confirmed totp_credential exists)
  → 200 {secret_base32, otpauth_uri}
  - Generates 160-bit secret (crypto/rand), base32-encodes without padding.
  - AES-256-GCM encrypts under current DEK with AAD 'totp:'+account_id+':'+key_version.
  - Inserts totp_credential row with confirmed_at = NULL, last_step = 0.
  - Returns the base32 secret and the otpauth URI so the frontend can render the QR.
    No QR PNG is generated server-side — the frontend has 'qrcode' libraries.

POST /me/totp/verify {code}  (sudo required iff a confirmed totp_credential exists)
  → 200 {recovery_codes: [...]}
  - Verifies the code via the standard ±1-step path.
  - On success: stamps confirmed_at, mints 10 recovery codes in the same
    transaction, returns them in the response body.
  - On failure: returns 401, totp throttle row updated. totp_credential row
    is left with confirmed_at = NULL; the next /me/totp/begin OVERWRITES
    (UPSERT) the row with a fresh secret.
```

**Recovery code format:** 10 characters from RFC 4648 base32 alphabet
(uppercase A-Z + 2-7, no padding), grouped as `XXXXX-XXXXX` with a
literal hyphen for legibility. ~50 bits entropy per code; 10 codes per
set; single-use; argon2id-hashed at rest in PHC format.

The literal `XXXXX-XXXXX` form is the over-the-wire format and what the
user sees / types. On verification, strip the hyphen and uppercase the
input before argon2id-comparing against each
`ListRecoveryCodesByAccount` row.

**Regeneration:** `POST /me/recovery-codes/regenerate` (sudo required)
deletes all rows via `DeleteAllRecoveryCodesByAccount`, mints 10 fresh
codes via `InsertRecoveryCode`, returns them in the response. Audit
event: `credential_event{factor:'recovery_code', event:'register'}` ×10.

References: Google Authenticator Key Uri Format spec
(github.com/google/google-authenticator/wiki/Key-Uri-Format); RFC 4648
(base32); RFC 9106 (argon2id).

### D5. WebAuthn-preferred policy via an explicit, anytime endpoint

`POST /api/prohibitorum/me/auth/revoke-password-totp` (sudo required)

Transactionally:

```sql
DELETE FROM password_credential WHERE account_id = $1;
DELETE FROM totp_credential     WHERE account_id = $1;
DELETE FROM recovery_code       WHERE account_id = $1;
```

Returns 204. Idempotent: calling on an already-passkey-only account is a
no-op and still returns 204.

Audit: three `credential_event` rows
(`factor:'password'|'totp'|'recovery_code', event:'revoke'`).

**UX integration (frontend, v0.6):** After
`POST /me/credentials/register/complete` adds a passkey, the frontend
checks `AvailableMethods(account)` and, if password+totp is still
enrolled, prompts: "Remove password and TOTP? Your passkey will become
your only sign-in method." Default **yes**. On confirmation the
frontend calls `/me/auth/revoke-password-totp` (sudo grant from the
just-completed passkey registration is fresh enough). The decision is
purely server-side enforced; v0.2 lays the endpoint and exposes it to
the (future) frontend.

This replaces the master spec's earlier wording of "Disable
password+TOTP backup? Default yes" — see Master Spec §"Factor policy
enforcement." The endpoint name `revoke-password-totp` is consistent
with the existing `/me/sessions/revoke` vocabulary in the codebase.

### D6. Sudo extended to three methods

Master spec v0.1 sudo re-proves a passkey via WebAuthn assertion. v0.2
extends to accept any factor the account has enrolled, so a
passkey-less account can still step up before sensitive operations.

**Method discovery:**

```
GET /api/prohibitorum/me/sudo/methods
  → 200 {methods: ["webauthn"|"password_totp"|"recovery_code"]}
```

Returns the subset of methods the current account can use, computed from
the rows in `webauthn_credential` / `password_credential`+
`totp_credential` / `recovery_code`. (`password_totp` requires BOTH a
password row and a confirmed TOTP row; either alone is insufficient.
`recovery_code` requires at least one unused row.) An account with no
methods returns an empty list (admin-recovery-only).

**Begin:**

```
POST /api/prohibitorum/me/sudo/begin {method: "webauthn"|"password_totp"|"recovery_code"}
  → 400 if method not in /sudo/methods response
  → 200 (webauthn): {publicKey: {challenge, ...}}    (existing shape)
  → 204 (password_totp): no challenge needed
  → 204 (recovery_code): no challenge needed
```

Stashes `{method, issued_at}` in KV under
`sudo_intent:<session_id>` with 5-min TTL. WebAuthn additionally stashes
the assertion ceremony state (existing v0.1 mechanism, unchanged shape).

**Complete:**

```
POST /api/prohibitorum/me/sudo/complete
  Body depends on stashed method:
    webauthn:        {assertion response}            (existing shape)
    password_totp:   {current_password, totp_code}
    recovery_code:   {recovery_code}
  → 204 on success: stamps SudoUntil = now + Auth.SudoTTL on the session
  → 401 on failure: increments auth_throttle for the relevant factor(s)
  → 429 if any relevant factor is currently locked
```

For `password_totp`, verify password first; if it fails, do NOT consult
TOTP (returns 401 immediately, password throttle bumped). If password
passes, verify TOTP code; on TOTP failure return 401 (TOTP throttle
bumped). Both must pass for the elevation.

For `recovery_code`, the code is consumed (marked `used_at`) on success
exactly like `/auth/recovery-code/verify` — sudo via a recovery code
burns a code.

**Sudo audit:** on success, emit
`credential_event{factor:'session', event:'sudo_granted',
detail:{method: "webauthn"|"password_totp"|"recovery_code"}}`.

**Sensitive endpoints requiring sudo:** All of these MUST call
`requireFreshSudo` before doing work.

```
v0.1 (unchanged):
  POST /me/credentials/register/begin
  POST /me/credentials/delete
  POST /me/credentials/rename
  POST /me/sessions/revoke

v0.2 (new):
  POST /me/password/set
  POST /me/totp/begin         (only if a confirmed totp_credential exists)
  POST /me/totp/verify        (only if a confirmed totp_credential exists)
  POST /me/recovery-codes/regenerate
  POST /me/auth/revoke-password-totp
```

The "only if a confirmed totp_credential exists" exception for
`/me/totp/begin|verify` covers the case of an account adding TOTP for
the first time: the user just authenticated, has no TOTP yet to sudo
with, and asking them for a different sudo method (e.g. password —
which they also might not have) before letting them set up MFA creates
a chicken-and-egg. Once any confirmed TOTP exists, swapping it requires
sudo.

For `/me/password/set` no such exception applies — a user setting a
password for the first time is performing a security-impacting action
and must sudo via WebAuthn (their existing method) or via
`password_totp` (their existing method). If they have neither, they
shouldn't be hitting this endpoint; that's an admin-recovery scenario.

## Implementation surface

### `pkg/credential/password/password.go`

Replace stubs with:

```go
type Store struct {
    q      db.Querier
    params configx.PasswordHashParams
    audit  audit.Writer
}

func NewStore(q db.Querier, params configx.PasswordHashParams, audit audit.Writer) *Store

// Set hashes via argon2id PHC and upserts.
func (*Store) Set(ctx, accountID, password) error

// Verify reads the row, argon2id-verifies, re-hashes on params upgrade,
// emits audit, updates auth_throttle. Returns ErrPasswordIncorrect on
// mismatch; ErrPasswordLocked (new) on lockout; ErrPasswordNotSet when
// no row.
//
// IMPORTANT: even when the row is missing, the caller (handle_auth_password.go)
// runs argon2id verify against dummyArgon2idHash to neutralize timing.
// Store.Verify itself only handles the row-present case; absence handling
// is one layer up.
func (*Store) Verify(ctx, accountID, password) error

// Delete removes the row (called by revoke-password-totp).
func (*Store) Delete(ctx, accountID) error
```

PHC format: `$argon2id$v=19$m=65536,t=3,p=4$<b64salt>$<b64tag>`.
Library: `golang.org/x/crypto/argon2` for the raw KDF; PHC
encode/decode is a ~30-line helper (no third-party PHC library
dependency — `crypto/subtle.ConstantTimeCompare` for the tag).

### `pkg/credential/totp/totp.go`

Replace stubs with:

```go
type Store struct {
    q      db.Querier
    deks   configx.DEKSet        // map[int][]byte
    cfg    configx.TOTPConfig
    audit  audit.Writer
}

// Begin generates secret, encrypts, inserts (UPSERT — overwrites a
// half-baked row from a prior unconfirmed attempt), returns
// {SecretBase32, ProvisioningURI}. Recovery codes are NOT yet minted.
func (*Store) Begin(ctx, accountID, username) (*Enrollment, error)

// Verify computes T, checks ±1-step drift, enforces last_step replay,
// updates last_step. On first success also stamps confirmed_at and
// mints 10 recovery codes (returned in the response). Emits audit.
// Honors auth_throttle.
func (*Store) Verify(ctx, accountID, code) ([]string /*recoveryCodes, nil on subsequent verifies*/, error)

// VerifyRecoveryCode argon2id-matches each recovery_code row in order,
// consumes the first match (used_at, used_session_id, used_ip). Returns
// nil on success, ErrRecoveryCodeInvalid on no match.
func (*Store) VerifyRecoveryCode(ctx, accountID, code, sessionID, ip) error

// RegenerateRecoveryCodes: tx { delete all, insert 10 new }. Returns
// the 10 plaintext codes (exactly once). Emits audit.
func (*Store) RegenerateRecoveryCodes(ctx, accountID) ([]string, error)

// Delete (called by revoke-password-totp): removes totp_credential row.
func (*Store) Delete(ctx, accountID) error
```

AES-GCM encryption uses the standard library `crypto/aes` +
`crypto/cipher.NewGCM`. AAD bytes: `"totp:" + accountIDString + ":" +
keyVersionString`. Nonce: 12 random bytes per row.

TOTP code computation: HMAC-SHA1 of 8-byte big-endian counter `T`, dynamic
truncation per RFC 4226 §5.3, modulo 10^digits. Library: implement
in-package (~40 lines) rather than pull a dep — algorithm is too small to
warrant the supply-chain surface.

### `cmd/smoke/main.go` extension

After the existing 17 steps + DB assertions, append (rough sequence —
the implementer can renumber for ergonomic flow):

```
18.  /me/sudo/begin webauthn → /me/sudo/complete (existing path; primes SudoUntil)
19.  /me/password/set {password: "smokepass1!"} → 204
20.  DB assert: password_credential row exists; hash starts with "$argon2id$v=19$"
21.  Wait 6 minutes, OR call a test-only helper that expires SudoUntil. Actually:
     do a new sudo via webauthn before each new sensitive op — straight-line.
22.  /me/sudo/begin webauthn → complete
23.  /me/totp/begin → parse otpauth_uri, base32-decode secret
24.  Compute current TOTP code, /me/totp/verify {code} → capture 10 recovery_codes
25.  DB assert: totp_credential.confirmed_at IS NOT NULL; 10 recovery_code rows
26.  /auth/logout (revoke session)
27.  /auth/password/begin {username, password} → partial_session_token
28.  Compute current TOTP code, /auth/totp/verify {partial_session_token, code} →
     session cookie set; /me round-trip
29.  /auth/logout
30.  /auth/password/begin → partial token
31.  /auth/recovery-code/verify {partial_session_token, code: recovery_codes[0]} →
     session cookie set; /me round-trip
32.  DB assert: recovery_code[0].used_at IS NOT NULL, used_session_id set
33.  /auth/logout
34.  Throttle observation: /auth/password/begin → partial token; submit /auth/totp/verify
     with a deliberately-wrong code. Repeat back-to-back until at least one 429+
     Retry-After is observed (typically by the 4th attempt — failures 1 and 2 are
     typo-grace 401s, failure 3 sets a 1s lockout, the next call within that lockout
     is 429). The harness does not assert an exact sequence — only "401s seen, then
     ≥1 429 seen, and DB shows auth_throttle row with failed_attempts ≥ 3 and
     locked_until > now."
35.  /auth/totp/verify with a correct code while still locked → expect 429 (lockout
     pre-empts the verify entirely).
36.  Reset for the rest of the smoke: smoke calls pkg/db directly to DELETE the
     auth_throttle row for this (account_id, 'totp'). This is a harness-only
     shortcut — production code only resets via successful verify. Document this
     in the smoke log line for clarity.
37.  /me/sudo/begin password_totp → 204
38.  /me/sudo/complete {current_password, totp_code} → 204; assert SudoUntil refreshed
39.  /me/sudo/begin recovery_code → 204
40.  /me/sudo/complete {recovery_code: recovery_codes[1]} → 204
41.  DB assert: recovery_codes[0] and [1] both have used_at set
42.  /me/sudo/begin webauthn → complete (priming for the destructive op)
43.  /me/auth/revoke-password-totp → 204
44.  DB assert: 0 rows in password_credential, totp_credential, recovery_code for this account
45.  /auth/password/begin {username, password} → 401 (password gone)
46.  DB assert: credential_event has rows for every register/use/fail/revoke/sudo_granted
     event we triggered above (count ≥ N for some sane N).
```

The smoke harness imports `pkg/credential/totp` directly for code
computation — it's the same package the server uses, so any bug in TOTP
code generation would surface symmetrically on both sides. We're
testing the storage, AAD, throttle, sudo, and HTTP wiring, not the
RFC 6238 arithmetic. (If we wanted to detect RFC 6238 algorithm bugs we
could shell out to `oathtool` for the smoke; that's overkill for v0.2
and adds a host dep.)

## Threat-model deltas

Beyond what the master spec already lists for password and TOTP:

- **Step-1 username enumeration via timing.** Mitigated by
  D3 (constant-time dummy verify on missing rows).
- **Step-1 step-2 split-flow attack.** A partial-session token is not a
  session — it grants nothing except the right to attempt step 2 once.
  Verified by smoke step 28 round-tripping `/me` only after step 2.
- **TOTP code replay within a 30-second window.** Mitigated by
  `last_step` per RFC 6238 §5.2 (existing v0.1 schema; v0.2 enforces).
- **Recovery-code redemption replay.** Mitigated by `used_at IS NOT
  NULL` predicate in `ConsumeRecoveryCode` (existing query; v0.2 wires).
- **Recovery-code mass leak.** Mitigated by argon2id at-rest hashing +
  per-account throttle + 50-bit entropy per code (10 codes × 2^50 =
  2^53.3 search space, throttled to 5/5min after the typo grace
  = ~134 million years at the cap to enumerate one code).
- **TOTP secret exfiltration via DB compromise.** Mitigated by AES-GCM
  encryption with AAD `'totp:'+account_id+':'+key_version` — copying
  ciphertext to a different row's `account_id` fails decryption.
- **Sudo-via-recovery-code stockpile attack.** A recovery code grants
  one elevation. After elevation, the code is consumed (it's the same
  consumption as the login path). An attacker who acquired a leaked
  recovery code can elevate once; then the code is gone. The
  `credential_event` `sudo_granted` row records `method:
  'recovery_code'` for the security team's anomaly detection.

## Open questions deferred

- **TOTP secret recompute on DEK rotation.** v0.7+ provides an admin
  tool that walks `totp_credential` rows where `key_version < current`,
  decrypts under the old key, re-encrypts under the new. v0.2 just
  honors `key_version` per row.
- **Conditional UI (passkey autofill) for username-first flows.**
  v0.7+; orthogonal to the password+TOTP design here.
- **Account-existence privacy on `/auth/methods?username=...`.** Defer
  to v0.6 when the login UX choice is concrete.
