# v0.5 Downstream SAML 2.0 IdP — Design Spec

**Status:** approved 2026-05-30 (brainstorming). Feeds the v0.5 implementation plan.

**Goal:** Ship Prohibitorum's downstream **SAML 2.0 Identity Provider** with a
**GitHub Enterprise Server (GHES)-compatible profile**: SP-initiated SSO
(HTTP-Redirect in, HTTP-POST out), IdP-local Single Logout, IdP metadata, a
metadata-ingesting `saml-sp` CLI, stable persistent NameID, the GHES attribute
profile, and a first-class XML-security hardening pass — end-to-end
smoke-verified against an in-process mock SP.

**Prereqs already in place (do not rebuild):** the full SAML schema landed in
`db/migrations/005_saml.sql` (`saml_sp`, `saml_sp_acs`, `saml_sp_key`,
`saml_subject_id`, `saml_session`) with sqlc queries generated in
`pkg/db/saml_sp.sql.go`; the skeleton `pkg/protocol/saml/saml.go`
(`IdP{q db.Querier}` + stubbed `Metadata`/`SSO`/`SLO`/`SubjectID` + sentinel
errors); the protocol-agnostic `signing-key` infra (RSA key + self-signed x509
cert by `kid`; `signing_key.x509_cert_pem` exists for SAML); `audit.FactorSAMLSP`.

**Reference:** `docs/superpowers/specs/2026-05-24-audit-saml.md` (the SAML audit —
GHES requirements, the new tables, code-level verdicts). This spec supersedes it
where they differ.

## Locked decisions

- **D1 — Hybrid library (not full samlidp).** Use `github.com/crewjam/saml`
  (SAML message/metadata types, marshaling) + `github.com/russellhaering/goxmldsig`
  (XML-DSig sign/verify, C14N) as **primitives only**. We own the handlers and
  drive everything from our DB-backed `saml_sp*` schema + the existing session
  store. We do **not** adopt `samlidp.Server` (its in-memory/Store model fights
  our schema and the GHES profile control). Mirrors the v0.4 OIDC philosophy
  (library for crypto, hand-rolled orchestration).
- **D2 — Scope = SP-initiated SSO + IdP-local SLO + metadata + CLI.**
  - **In:** SP-initiated SSO (the GHES login flow), IdP-local SLO (validate a
    signed LogoutRequest → revoke our session → signed LogoutResponse; **no**
    front-channel propagation to the user's other SPs), IdP `EntityDescriptor`
    metadata, `saml-sp {create,list}` CLI with metadata ingestion, stable
    persistent NameID, the GHES attribute profile.
  - **Out (deferred):** IdP-initiated SSO; front-channel multi-SP SLO; assertion
    **encryption** (GHES doesn't require it; `saml_sp_key.use='encryption'`
    schema room stays unused); AttributeQuery / NameIDMapping / Artifact binding.
- **D3 — `ForceAuthn` deferred, `IsPassive` honored minimally** (parity with the
  v0.4 deferral of OIDC `prompt=login`/`max_age`). An existing valid session
  satisfies an AuthnRequest (no forced re-auth). `IsPassive=true` + no session →
  a SAML `urn:oasis:names:tc:SAML:2.0:status:NoPassive` error Response (the SAML
  analog of `login_required`), not a login-UI bounce.
- **D4 — Sign BOTH `<Response>` and `<Assertion>`** (RSA-SHA256 + exclusive
  C14N). The safe superset of GHES's "sign Response and/or Assertion"; GHES
  validates `Destination` only when the Response is signed.
- **D5 — Verify SP signatures against the registered cert only.** SP-signed
  AuthnRequests/LogoutRequests are verified against the cert(s) in `saml_sp_key`
  (ingested from SP metadata), **never** a cert embedded in the incoming message.
  This is both correct trust-anchoring and a sidestep of the known
  goxmldsig/crewjam cert-chain pitfall (crewjam/saml#384).
- **D6 — Stable opaque NameID** per `(account, sp)` in `saml_subject_id`:
  generated as 32-byte `crypto/rand` base64url on first SSO, reused forever
  (GHES re-link safety; Core §8.3.7). Default format
  `urn:oasis:names:tc:SAML:1.1:nameid-format:persistent` (the 1.1 URI).
- **D7 — Reuse the OIDC signing-key infra.** The same active `signing_key` RSA
  key signs SAML; SAML uses its `x509_cert_pem`. Reuse `parseRSAPrivatePEM` + the
  key-cache pattern from `pkg/protocol/oidc/keys.go`. IdP metadata publishes
  every signing cert where `retired_at IS NULL OR retired_at > now() - grace`
  (audit R6).
- **D8 — Issuer/EntityID = `PUBLIC_ORIGIN`.** The IdP `entityID` and endpoint
  URLs derive from `configx` `PublicOrigins[0]` (same source as the OIDC issuer):
  metadata `…/saml/metadata`, SSO `…/saml/sso`, SLO `…/saml/slo`.
- **D9 — Per-identity rate limiting** (consistent with v0.3/v0.4, not per-IP):
  reuse `s.rateLimiter` with keys like `saml:sso:sp:<entity_id>` and
  `saml:sso:acct:<id>`.

## Architecture

A hand-rolled `IdP` in `pkg/protocol/saml` owns the handlers, widened from the
skeleton to carry its dependencies:

```
IdP{ cfg, queries db.Querier, kv kv.Store, sessions *session.SessionStore,
     audit audit.Writer, rl *authn.RateLimiter, keys *<signing-key cache> }
NewIdP(cfg, queries, kv, sessions, audit, rl) *IdP
```

`crewjam/saml` + `goxmldsig` are promoted to direct deps (`go get` + `go mod
tidy`). The HTTP surface, mounted in `pkg/server/server.go`:

| Route | Purpose |
|---|---|
| `GET /saml/metadata` | IdP `EntityDescriptor` (signing certs, SSO/SLO endpoints, NameIDFormat, `WantAuthnRequestsSigned`) |
| `GET\|POST /saml/sso` | SP-initiated SSO — parse+validate AuthnRequest, session-gate, signed Response+Assertion auto-POSTed to the SP ACS |
| `GET\|POST /saml/slo` | IdP-local SLO — validate signed LogoutRequest, revoke session, signed LogoutResponse |

`/saml/sso` and `/saml/slo` run behind the existing global `LoadSession`
middleware (so `authn.SessionFromContext` works), like `/oauth/authorize`.

### Files (`pkg/protocol/saml/`)

| File | Responsibility |
|---|---|
| `idp.go` | widened `IdP` struct + `NewIdP`; shared helpers |
| `metadata.go` | render IdP `EntityDescriptor`; **parse SP metadata** for CLI ingestion |
| `authnreq.go` | parse + validate inbound AuthnRequest (decode, sig-verify, replay, Destination/ACS) |
| `assertion.go` | build + sign Assertion + Response (subject/conditions/authnstatement) |
| `attributes.go` | project `account` → SAML attributes per `attribute_map` (GHES profile) |
| `subjectid.go` | get-or-create stable opaque NameID (`SubjectID`) |
| `sso.go` | `HandleSSO` orchestration (replaces stub) |
| `slo.go` | `HandleSLO` orchestration (replaces stub) |
| `xmlsec.go` | hardened XML/DSig layer: DTD/XXE-off parsing, SHA-256-only verify, reference-ties-to-element + duplicate-ID rejection, exclusive-C14N config |
| (CLI) `cmd/prohibitorum/main.go` | `saml-sp {create,list}` (metadata ingestion + overrides) |
| (wiring) `pkg/server/server.go` | construct `NewIdP(...)`, mount the 3 routes |

**Reuse:** `signing_key` key + cert, `parseRSAPrivatePEM` + key-cache pattern,
the shared session store / audit / rate limiter / KV, `authn.SessionFromContext`.

## Data flow

### SP-initiated SSO
1. SP redirects browser to `GET /saml/sso?SAMLRequest=<deflate+base64>&RelayState=…`
   (+ `SigAlg`+`Signature` query params when signed — HTTP-Redirect binding signs
   the query string, detached).
2. Decode/parse the AuthnRequest → `GetSAMLSPByEntityID(Issuer)`. Unknown SP →
   **direct error** (never redirect to an untrusted target).
3. If the SP requires signed AuthnRequests (`require_signed_authn_request` /
   metadata `AuthnRequestsSigned`) → verify the redirect-binding signature against
   the SP's `saml_sp_key` signing cert(s); reject SHA-1 / absent.
4. Validate: AuthnRequest `ID` **single-use** (KV replay cache, TTL ~ request
   window); `Destination` = our SSO URL; requested ACS URL (if present) **exact-
   matches** a `saml_sp_acs.location` else use the SP's default ACS (open-redirect
   guard).
5. Session gate (`authn.SessionFromContext`): nil → 302 to
   `Issuer+/login?return_to=<SSO URL>`; nil+`IsPassive` → `NoPassive` Response;
   `ForceAuthn` ignored (D3). Reject the disabled-account sentinel (`Data==nil` /
   `Account.Disabled`) — lesson from the v0.4 deep audit.
6. NameID: `GetSAMLSubjectID(account, sp)` → if absent generate + `InsertSAMLSubjectID`.
7. Build + sign the Assertion (Subject incl. `SubjectConfirmationData`
   Recipient=ACS / NotOnOrAfter / InResponseTo; `Conditions` Audience=`entity_id`
   verbatim, NotBefore/NotOnOrAfter ±180s; `AuthnStatement`
   AuthnContextClassRef=PasswordProtectedTransport (stronger when WebAuthn),
   Comparison=exact, SessionIndex, SessionNotOnOrAfter from `session_lifetime`),
   wrap in a signed Response (Destination=ACS, InResponseTo, Status=Success).
8. Persist a `saml_session` row (for SLO). Audit `FactorSAMLSP` use.
9. Render an auto-POST HTML form to the ACS with `SAMLResponse` (base64) + echoed
   `RelayState`.

### IdP-local SLO
1. SP sends a signed LogoutRequest to `/saml/slo` (redirect or POST).
2. `GetSAMLSPByEntityID` + verify signature + validate (Destination, NotOnOrAfter,
   NameID resolves to a `saml_session`).
3. Revoke the Prohibitorum session (the session tied to the `saml_session`);
   delete the `saml_session` rows.
4. Build + sign a `LogoutResponse` (Status=Success); return via the SP's SLO
   response location. Audit `logout`.

## GHES attribute profile

`attribute_map` is an **ordered JSONB array** (audit R1); each entry
`{name, name_format, friendly_name?, source, multi}` — `source` names an
`account` column or `account.attributes` JSONB key; `multi:true` emits one
`<AttributeValue>` per array element. The GHES-default map (applied by
`saml-sp create --kind ghes`):

| SAML `Name` | NameFormat | source | multi |
|---|---|---|---|
| `USERNAME` (configurable FriendlyName) | basic | `username` | no |
| `administrator` (literal; **not** configurable — audit #5) | basic | `attributes.administrator` or `role=='admin'` → `"true"` | no |
| `emails` | basic | `attributes.emails` | yes |
| `public_keys` (Name `urn:oid:1.2.840.113549.1.1.1` — audit #7) | uri | `attributes.public_keys` | yes |
| `gpg_keys` | basic | `attributes.gpg_keys` | yes |

NameFormat URIs: basic = `urn:oasis:names:tc:SAML:2.0:attrname-format:basic`,
uri = `…:attrname-format:uri` — unit-tested to avoid namespace drift. Non-GHES
SPs default to a minimal map (NameID + a configurable username attribute) and can
override the whole map via the CLI.

## Security hardening (first-class — drives dedicated plan tasks)

The SAML attack surface is dominated by **XML Signature Wrapping (XSW)** and XML
parser abuse. We are the IdP but we **verify SP-signed AuthnRequests +
LogoutRequests**, so these defenses apply to our verification path. Sources:
[OWASP SAML Security Cheat Sheet], [PortSwigger "The Fragile Lock"],
[WorkOS "SAML's signature problem"], [crewjam/saml#384], [GHES 3.17 SAML config
reference].

- **Sign both** Response + Assertion; RSA-**SHA256** + exclusive C14N
  (`MakeC14N10ExclusiveCanonicalizerWithPrefixList`). crewjam/goxmldsig fixtures
  default to RSA-SHA1 — we set the signature method explicitly and **reject
  SHA-1** on verify.
- **XSW:** the signature `<Reference>` URI MUST resolve to the exact element we
  consume; **reject duplicate IDs**; process only the signed element.
- **XXE:** DTD processing and external-entity resolution **disabled**;
  trusted-local-schema validation only (no remote schema fetch).
- **Cert pinning (D5):** verify against `saml_sp_key`, never message-embedded
  certs.
- **Replay + windows:** AuthnRequest/LogoutRequest `ID` single-use in KV;
  validate `Destination`; `NotBefore`/`NotOnOrAfter` with ±180s skew
  (shibboleth/crewjam `MaxClockSkew` default).
- **Open-redirect guard:** ACS URL exact-match against `saml_sp_acs`; unknown SP →
  direct error, never a redirect.
- `AudienceRestriction` = `entity_id` verbatim; `Recipient` = ACS;
  `InResponseTo` echoed.
- Per-SP / per-account rate limiting (D9). Audit `FactorSAMLSP` on success **and**
  failure (failed signature / audience mismatch / replay).
- NameID opaque + stable (D6).

[OWASP SAML Security Cheat Sheet]: https://cheatsheetseries.owasp.org/cheatsheets/SAML_Security_Cheat_Sheet.html
[PortSwigger "The Fragile Lock"]: https://portswigger.net/research/the-fragile-lock
[WorkOS "SAML's signature problem"]: https://workos.com/blog/saml-signature-problem
[crewjam/saml#384]: https://github.com/crewjam/saml/issues/384
[GHES 3.17 SAML config reference]: https://docs.github.com/en/enterprise-server@3.17/admin/managing-iam/iam-configuration-reference/saml-configuration-reference

## Testing

- **Unit** per component: metadata render; AuthnRequest parse + signature verify;
  **assertion build → round-trip verify** (the key interop test — sign a Response,
  then parse it back with **crewjam/saml's SP-side `ParseResponse`** against our
  IdP metadata: proves a real SP accepts what we emit); attribute projection;
  NameID stability (two calls → identical value); replay rejection;
  signed-AuthnRequest enforcement.
- **XSW/XXE negatives:** wrapped / duplicate-assertion / duplicate-ID /
  DTD-bearing payloads → assert rejection.
- **`cmd/smoke` mock-SP** (in-process, mirroring the v0.4 mock-RP): generate a
  signed AuthnRequest → drive `/saml/sso` with an authenticated session → receive
  + verify the POSTed Response (signature vs `/saml/metadata`, NameID,
  Recipient=ACS, Audience, attributes) → drive `/saml/slo`. DB asserts:
  `saml_subject_id` stable across two SSOs, `saml_session` row written,
  `credential_event` `factor='saml_sp'`. Negatives: unsigned AuthnRequest when
  required → reject; bad ACS → reject; replayed AuthnRequest ID → reject.

`cmd/smoke` runs all v0.2+v0.3+v0.4+v0.5 steps green against live PG + fresh dev
server + in-process mock SP; docs updated (STATUS/AUDIT/INTEGRATION) with honest
smoke-covered / unit-tested-only / deferred lists. After landing, a
post-implementation parallel audit (crypto/XML-DSig + protocol/standards +
race/logic) + a deep second pass (integration / data-integrity / schema-drift),
same discipline as v0.3/v0.4 — with **special focus on XSW/XXE, signature
verification, NameID stability, and replay**.
