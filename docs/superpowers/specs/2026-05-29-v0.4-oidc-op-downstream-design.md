# v0.4 — Downstream OIDC OP — design spec

> Delta on `docs/superpowers/specs/2026-05-24-multi-protocol-rescope-design.md`.
> Conformance checklist lives in `AUDIT.md` §"OIDC OP downstream". This spec
> records the decisions, architecture, storage model, per-endpoint flows, and
> verification gates for the v0.4 implementation.

## Goal

Ship Prohibitorum's downstream OpenID Connect Provider: the full first-party
OP surface so internal relying parties can authenticate users against the IdP
via the Authorization Code + PKCE flow. Endpoints:

- `GET /oauth/authorize` — auth-code issuance (session-gated).
- `POST /oauth/token` — `authorization_code` + `refresh_token` grants, with
  refresh rotation, reuse detection, and family revocation.
- `GET|POST /oauth/userinfo` — claim projection for a bearer access token.
- `POST /oauth/introspect` — RFC 7662, first-party RPs.
- `POST /oauth/revoke` — RFC 7009.
- `GET /oidc/logout` — RP-Initiated Logout 1.0.
- `GET /oauth/jwks` — real key set (replaces the v0.1 empty stub).
- `GET /.well-known/openid-configuration` — expanded discovery doc.
- `prohibitorum signing-key generate` and `prohibitorum oidc-client {create,list}` CLIs.

**Tech stack:** Go 1.26, `go-jose/v4` (JWT sign/verify — promoted from indirect
to direct), `golang.org/x/crypto/argon2` (client secret hashing, reusing the
v0.2 helper), pgx/v5 via sqlc, chi, the existing KV store, session store,
audit writer, and account/client-keyed rate limiter.

**Approach (B, hand-rolled):** thin chi-mounted handlers owned by the
`Provider` in `pkg/protocol/oidc`, matching the project's established pattern
(v0.2 / v0.3 / `cmd/smoke/mockop` are all hand-rolled with narrow library use).
The `zitadel/oidc/v3` `pkg/op` *framework* is **not** adopted — it would invert
control and force our bespoke session/sudo/audit model to adapt to its Storage
interface. We may borrow a handful of `pkg/oidc` constant names where it is a
pure win, with no struct coupling.

## Decisions

- **D1 — Full surface, one spec.** All endpoints + both CLIs land in v0.4 as a
  single cohesive design, keeping token/claim/key invariants consistent.
- **D2 — Consent: auto-approve + reserved flag.** Configured `oidc_client` rows
  are trusted; `/authorize` auto-approves once a valid session exists. A new
  `oidc_client.require_consent boolean DEFAULT false` is reserved: when `true`,
  `/authorize` returns `consent_required` (no consent UI until v0.6).
- **D3 — Rate limits keyed on identity, not IP.** `/authorize` per `account_id`;
  `/token`, `/introspect`, `/revoke` per `client_id`; `/userinfo` per `sub`.
  Reuses the existing account/session-keyed limiter. Honors both the STATUS.md
  rate-limit goal and the v0.3 M5 decision that client IP is untrustworthy.
- **D4 — Client provisioning via CLI.** `oidc-client create` generates +
  argon2id-hashes the secret and prints it once. HTTP admin CRUD stays deferred
  to the v0.6 frontend.
- **D5 — Hand-rolled OP + `go-jose/v4`.** See Approach above.
- **D6 — Opaque `sub` via a dedicated column.** `account.oidc_subject uuid NOT
  NULL UNIQUE DEFAULT gen_random_uuid()` (Postgres 18 built-in). DB-side default
  means no account-creation path changes; sqlc regenerates the `Account` model
  and `claims.go` reads it. Pairwise subject salting stays deferred to v0.7+.
- **D7 — No-session `/authorize` → 302 to login.** Redirect to
  `OIDC.Issuer + /login?return_to=<authorize URL>` (login page is a v0.6
  deliverable; the redirect *contract* is correct now). `prompt=none` + no
  session → `login_required` without redirect.
- **D8 — Storage split (load-bearing):**
  - Authorization codes → **KV**, single-use, replay → family revocation.
  - Refresh tokens → **KV**, opaque, rotated, with a per-family record for
    reuse detection. The refresh-family *forensics table* stays deferred
    (KV-only for v0.4, per AUDIT.md).
  - Access tokens → **self-contained RFC 9068 JWT**, not stored; revocation via
    the existing `revoked_jti` PG denylist.
  - ID tokens → self-contained JWT, not stored.

## Storage model detail

### Authorization codes (`codes.go`)

- Key `oidc:code:<code>`, value JSON `{client_id, account_id, session_id,
  redirect_uri, scope, nonce, code_challenge, code_challenge_method, auth_time,
  amr, acr}`, TTL = `OIDC.AuthorizationCodeTTL`.
- Mint: random 32-byte base64url code; `SetEx`.
- Consume (at `/token`): `Pop` (atomic single-use → concurrent double-submit
  yields exactly one winner). On success, write a used-marker
  `oidc:code:used:<code>` → `{family_id}` (or `{}` when no refresh issued), TTL
  slightly longer than the code TTL.
- Replay: `Pop` misses → check `oidc:code:used:<code>`. Present → **replay**:
  revoke the recorded refresh family, audit `reason=code_replay`, return
  `invalid_grant`. Absent → `invalid_grant` (unknown/expired).

### Refresh tokens (`refresh.go`)

- Family record `oidc:refresh:family:<fid>` → `{current_token, client_id,
  account_id, session_id, scope, auth_time, amr, acr, issued_at}`, TTL =
  `OIDC.RefreshTokenTTL` (sliding — refreshed on each rotation).
- Token map `oidc:refresh:<token>` → `{family_id}`, TTL = `RefreshTokenTTL`.
- Presentation: `token → family_id → family`. If `family` missing →
  `invalid_grant` (revoked/expired). If `token == family.current_token` →
  **rotate**: mint new token, set `family.current_token = new`, write the new
  token map. If `token != family.current_token` → **reuse of a superseded
  token**: delete the family record, audit `reason=refresh_reuse`, return
  `invalid_grant` (old token maps self-expire; their family lookup now fails).

### Access tokens

- RFC 9068 JWT, never stored. Revocation: `jti` written to `revoked_jti` on
  `/oauth/revoke` (access-token hint) and on family revocation where a recent
  access `jti` is known. `/userinfo` + `/introspect` consult `IsJTIRevoked`.

## Token formats

### ID token (OIDC Core §2), TTL = `OIDC.IDTokenTTL`

`iss, sub (=account.oidc_subject), aud (=client_id), exp, iat, auth_time,
nonce` (when present in the authz request), `amr`, `acr`, `sid (=session.id)`,
`at_hash` (left-half SHA-256 of the access token, base64url), `azp` (emitted
only when `aud` is multi-valued — single-client today). Profile claims
(`username, displayName, role, attributes`) folded in **only when the `profile`
scope is granted**. JOSE header `{alg:RS256, typ:JWT, kid}`.

### Access token (RFC 9068), TTL = `OIDC.AccessTokenTTL` (default 10m)

JOSE header `{alg:RS256, typ:at+jwt, kid}`; claims `iss, sub, aud, exp, iat,
jti` (random), `client_id`, `scope`.

## Endpoint flows

### `GET /oauth/authorize`

1. Load `client_id` → `oidc_client` (exists, `disabled=false`). Validate
   `redirect_uri` by **exact match** against `client.redirect_uris`. Invalid
   client or redirect_uri → render an error page directly; **never redirect to
   an unvalidated URI**.
2. Validate, redirecting recoverable errors to `redirect_uri?error=…&state=…`:
   `response_type=code` only (else `unsupported_response_type`); `scope ⊆
   allowed_scopes`, must include `openid` (else `invalid_scope`); PKCE
   `code_challenge` required, `code_challenge_method ∈
   allowed_code_challenge_methods` (S256; `plain` rejected) else
   `invalid_request`; capture `state`, `nonce`.
3. Session check via the existing middleware. Authenticated → rate-limit per
   `account_id`. No session → 302 to `Issuer + /login?return_to=<authorize
   URL>` (D7); `prompt=none` + none → `login_required` (no redirect);
   `prompt=login` or `max_age` exceeded by `session.auth_time` → force re-auth
   via the same redirect.
4. `require_consent=true` → `consent_required` error (D2). Default → continue.
5. Mint code into KV; 302 to `redirect_uri?code=…&state=…&iss=<Issuer>`
   (RFC 9207). Audit `reason=authorize`.

### `POST /oauth/token`

Client authentication (`client.go`) per `token_endpoint_auth_method`:
`client_secret_basic` (Authorization header), `client_secret_post` (form), or
`none` (public; PKCE mandatory). argon2id verify for confidential. Bad client
auth → `invalid_client` (401, `WWW-Authenticate` for basic). Rate-limit per
`client_id`. Dispatch on `grant_type`:

- **`authorization_code`**: consume code (+ replay handling above); assert
  `code.client_id == authenticated client`; assert `redirect_uri` equals the
  code's; **PKCE verify** `base64url(SHA256(code_verifier)) == code_challenge`;
  mint access + ID token; iff `offline_access` granted, a refresh token (new
  family); write the used-marker with the `family_id`. Respond `{access_token,
  token_type:"Bearer", expires_in, id_token, refresh_token?, scope}`. Audit
  `reason=token_issued`.
- **`refresh_token`**: look up → family → reuse detection (rotate or
  revoke-family); assert `family.client_id == authenticated client`; **re-fetch
  account, reject if `disabled`** (`invalid_grant`); mint fresh access token +
  rotated refresh token + re-issued ID token. Audit `reason=refresh_rotated`.
- Unknown grant → `unsupported_grant_type`.

### `GET|POST /oauth/userinfo`

Bearer access token → verify (sig by `kid`, `iss`, `exp`, `typ:at+jwt`,
`IsJTIRevoked`). Resolve `account` by `oidc_subject`; reject if `disabled`.
Return `{sub}` + profile claims gated by the token's `scope`. Rate-limit per
`sub`. `WWW-Authenticate: Bearer` with `invalid_token` on failure.

### `POST /oauth/introspect` (RFC 7662)

Client-authenticated. Access-token JWT → verify + `IsJTIRevoked` → `{active:true,
sub, scope, client_id, exp, iat, token_type:"Bearer"}`. Refresh token → family
lookup → `{active:true, ...}` or `{active:false}`. Anything
invalid/expired/revoked → `{active:false}` (no detail leak). Audit.

### `POST /oauth/revoke` (RFC 7009)

Client-authenticated. Access token → `InsertRevokedJTI(jti, exp)`. Refresh token
→ delete its family (revokes all descendants; outstanding access tokens
self-expire ≤ `AccessTokenTTL`, permitted by RFC 7009). Always `200`, even for
unknown tokens. Audit `reason=revoked`.

### `GET /oidc/logout` (RP-Initiated Logout 1.0)

Validate `id_token_hint` (a token we issued; signature + `iss`) → extract `sid`
+ `aud`. Revoke the Prohibitorum session via `sessionStore` keyed on `sid`
(SSO sign-out). Validate `post_logout_redirect_uri` by exact-match against the
client's `post_logout_redirect_uris`; 302 there with `state` (or to a default
logged-out landing when omitted). Audit `reason=logout`. Front/back-channel
logout to other RPs is deferred (v0.7+).

## Cross-cutting

- **Discovery doc** (`oidc.go`): advertise `scopes_supported`
  `[openid, profile, offline_access]`, `introspection_endpoint`,
  `revocation_endpoint`, `end_session_endpoint`, real `claims_supported`,
  `code_challenge_methods_supported [S256]`,
  `authorization_response_iss_parameter_supported: true`,
  `token_endpoint_auth_methods_supported`. `Cache-Control` from `JWKSCacheMaxAge`.
- **Errors** (`errors.go`): `oidcError(error, description)` → RFC 6749 §5.2 JSON
  for token-family endpoints; query-param errors for `/authorize`. Distinct
  vocabulary from `authn.AuthError` (OAuth error codes). HTTP statuses per spec
  (`invalid_client` → 401; most others → 400).
- **Audit**: factor `oidc`. Structured `reason` per event: `authorize`,
  `token_issued`, `refresh_rotated`, `refresh_reuse`, `code_replay`, `revoked`,
  `logout`, and failure reasons (`invalid_client`, `invalid_grant`,
  `account_disabled`, …). Every failure path emits `EventFail` with a reason;
  successes emit `EventUse` / structured events.
- **Rate limits** (D3): keys `oidc:authorize:acct:<id>`,
  `oidc:token:client:<id>`, `oidc:introspect:client:<id>`,
  `oidc:revoke:client:<id>`, `oidc:userinfo:sub:<sub>`. Sensible per-minute
  caps (RPs are machines → higher than human-facing limits).

## Testing

- **Unit tests** per file: code mint/consume/replay; refresh
  rotation + reuse→family-revoke; JWT sign/verify + `kid` resolution + `alg:none`
  rejection; claims scope-gating; client auth across all three methods; PKCE
  verify; each handler's happy + error branches.
- **Mock RP in `cmd/smoke`** (mirrors `cmd/smoke/mockop`): generate a signing
  key + register a client via the CLIs, then drive the full flow against live
  PG + dev server — `/authorize` (with a pre-established WebAuthn session) →
  code → `/token` → validate ID + access tokens → `/userinfo` → `/introspect`
  → refresh-rotate → **reuse detection (replay the superseded refresh ⇒ family
  revoked, `invalid_grant`)** → `/revoke` → `/oidc/logout`. ~15–20 steps with DB
  + KV state assertions (`revoked_jti` rows, `credential_event` factor=`oidc`).
- **Post-implementation 3-auditor pass** (crypto + protocol/standards +
  race/logic) against the v0.4 commit range, same discipline as v0.2/v0.3.

## Deferred (out of v0.4 scope)

Per AUDIT.md: PAR (RFC 9126), JAR (RFC 9101), DPoP (RFC 9449), mTLS (RFC 8705),
dynamic client registration (RFC 7591), pairwise `sub`, encrypted ID tokens
(JWE), front/back-channel logout, the refresh-family forensics table, and the
consent UI (v0.6 frontend).
