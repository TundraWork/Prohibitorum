# v0.3 — Upstream OIDC Federation design

**Status:** approved 2026-05-28, ready for implementation.

This document is a **delta on top of** the v0.1 multi-protocol rescope spec
(`docs/superpowers/specs/2026-05-24-multi-protocol-rescope-design.md`) and
its embedded v0.3 plan. The master spec defines the schema (`upstream_idp`,
`account_identity`), AAD format for `client_secret_enc`, three provisioning
modes, mix-up resistance via snapshotted `expected_iss`, and the HTTP
endpoints. Read it first. This delta locks down the ambiguities the
master spec left open and adds the items the audit-driven discipline
from v0.2 requires.

## Scope

v0.3 implements upstream OIDC federation as a sign-in path:

1. Per-IdP RP flow via `zitadel/oidc/v3` with PKCE S256.
2. Three provisioning modes: `auto_provision`, `invite_only`, `link_only`.
3. `account_identity` linkage keyed on `(upstream_iss, upstream_sub)` per
   OIDC Core §2.
4. `/me/identities` list + unlink (with last-sign-in-method safety).
5. `/me/identities/link/{slug}/{begin,callback}` self-service link flow
   for a logged-in user to add an upstream IdP.
6. Session `amr` propagated verbatim from upstream ID token (empty if
   upstream omits the claim).
7. `cmd/smoke` extended with ~15 federation steps driven against an
   in-process mock OP.
8. Admin `upstream_idp` CRUD via direct SQL only (HTTP admin endpoints
   deferred to v0.6 frontend).

## Non-goals

- No upstream refresh-token storage. Local session lifetime is independent
  of the upstream session. (Federation is login-only, not a continuous
  trust relationship.)
- No HTTP admin endpoints for `upstream_idp` CRUD; ops manages IdPs via
  direct DB writes in v0.3. v0.6 dashboard ships the admin UI.
- No frontend changes (`dashboard/` is empty; v0.6).
- No SAML federation (that's downstream IdP work in v0.5).
- No automatic upstream-DEK rotation tool; the `key_version` column
  supports rotation manually (v0.7+).
- No advanced flows (PAR, DPoP, JAR, mTLS); not required for v0.3.

## Decisions resolved by this delta

### D1. Per-IdP `require_verified_email` (schema addition)

OIDC Core §5.1 defines `email_verified` as a boolean claim. Permissive
upstreams that accept unverified email registrations are the main
JIT-account-squatting vector. Strict default with per-IdP override.

**Schema:** new column on `upstream_idp`:

```sql
ALTER TABLE upstream_idp
  ADD COLUMN require_verified_email boolean NOT NULL DEFAULT true;
```

Land via either:
- A new migration `db/migrations/006_federation_v03.sql` (clean history),
  OR
- Amend `db/migrations/004_federation.sql` in place (pre-deployment, per
  `feedback_picotera_decoupling.md`).

Implementer picks based on the project's deployment-readiness state at
the time of implementation. Either is fine.

**Behavior in `auto_provision` mode:**

When `require_verified_email = true` (default) AND the upstream ID token's
`email_verified` claim is absent or false: reject with HTTP 403, error
code `email_not_verified`. Audit `factor:federation_oidc, event:fail,
detail:{reason:"email_not_verified"}`. Account is NOT provisioned.

When `require_verified_email = false`: skip the check.

`invite_only` and `link_only` modes don't gate on `email_verified` — the
admin has already authorized the link via enrollment token or the user
is linking to their existing account.

### D2. Re-login claim sync

On every successful federation callback (after `account_identity` is
matched or created), the handler updates:

- `account.display_name` from upstream `display_name_claim` value (if
  changed).
- `account_identity.upstream_email` from upstream `email_claim` value
  (if changed).

The username field (account.username) is NEVER updated post-creation —
it's the immutable local identifier. Renames go through admin.

**Why update on every login:** keeps local fields in sync with the IdP's
source of truth. A user who renames themselves at the IdP sees the change
reflected without admin intervention. A compromised upstream that swaps a
display name doesn't move account ownership — username (which is the
identity) is unchanged; the attacker still has to compromise the
`(upstream_iss, upstream_sub)` mapping which `account_identity` already
defends against.

**Edge case:** if the user changed their local display_name out-of-band
(e.g., admin renamed them), the next federation login overwrites it.
Document as known behavior; v0.6+ may add a "lock local display_name"
flag if requested.

### D3. `/me/identities/{id}/unlink` last-sign-in-method safety

The unlink endpoint enforces server-side that removing this identity
does NOT leave the account with zero usable sign-in methods.

**Check:** before deleting the `account_identity` row, compute:

```
remaining_methods = (
    count(webauthn_credential WHERE account_id = $1) > 0,
    (count(password_credential WHERE account_id = $1) > 0 AND
     count(totp_credential WHERE account_id = $1 AND confirmed_at IS NOT NULL) > 0),
    count(account_identity WHERE account_id = $1 AND id != $2) > 0,
)
```

If all three are false → reject with HTTP 400, error code
`last_sign_in_method`, suggesting "Add another sign-in method via
/me/credentials or /me/totp before unlinking this identity."

If any one is true → proceed with delete.

**Implementation:** extend `pkg/authn/flow.go:AvailableMethods` to also
consider `account_identity` rows. Add a new `MethodFederationOIDC` to
the returned slice when ≥1 identity exists. The unlink handler then
projects "what methods would remain if I removed identity $2" and
rejects if empty.

This is sudo-gated. The user must have a fresh sudo grant via webauthn
or password+totp before unlinking (consistent with other sensitive
`/me/*` operations).

### D4. Federated session `amr`

Pass upstream's ID-token `amr` claim through verbatim into the local
session row. If upstream omits the claim, the local session's `amr` is
empty `[]`.

**Rationale:** RFC 8176 doesn't define `federated`. Inventing values
misleads downstream RPs that read `amr` for policy decisions. Empty
`amr` is RFC-permitted. Downstream consumers gating on "user used
federation" can also key off `session.upstream_idp_id` which already
exists in the schema (set by this v0.3 work).

**Implementation:** `pkg/server/handle_federation.go:handleFederationCallbackHTTP`
extracts `amr` from the verified ID token claims and passes the
slice to `sessionStore.Issue(ctx, accountID, ip, ua, amr)`.

### D5. State KV separation between login flow and link flow

Two distinct KV prefixes:

- `oidc:fed:state:<random>` — minted at `/auth/federation/{slug}/login`,
  consumed at `/auth/federation/{slug}/callback`. The flow that issues a
  session.
- `oidc:fed:link:<random>` — minted at
  `/me/identities/link/{slug}/begin`, consumed at
  `/me/identities/link/{slug}/callback`. The flow that links to the
  current session's account WITHOUT issuing a new session.

KV value shape is the same in both cases:

```json
{
  "idp_id": <bigint>,
  "expected_iss": "<snapshot>",
  "expected_token_endpoint": "<snapshot>",
  "nonce": "<random>",
  "code_verifier": "<RFC 7636 PKCE>",
  "return_to": "<post-flow URL>",
  "linking_account_id": <bigint or null>  // populated only for link flow
}
```

The link callback validates that the current request's session belongs
to `linking_account_id` (defense against session-swap mid-flow).

Both keys are consumed via `kv.Store.Pop` (atomic, single-use) — same
primitive added in v0.2 audit Bundle 1.

### D6. `return_to` validation (open-redirect defense)

`return_to` query parameter on `/auth/federation/{slug}/login` and
`/me/identities/link/{slug}/begin` is validated against an allowlist:

- Empty or absent → default to `/` (root of `PROHIBITORUM_PUBLIC_ORIGIN`).
- Relative path (begins with `/` but not `//`) → accepted as
  same-origin path.
- Absolute URL → must match `PROHIBITORUM_PUBLIC_ORIGIN` scheme+host
  exactly (string compare, not URL-parsed compare — defense against
  parser-confusion).

On rejection: return 400 with `invalid_return_to` error code. Do NOT
attempt the federation flow.

### D7. Mix-up + replay defenses

Per RFC 9700 OAuth 2.0 Security BCP and RFC 9207 issuer-parameter:

1. **`expected_iss` snapshot:** at request time, fetch the IdP's
   discovery doc and snapshot `issuer` AND `token_endpoint` into the
   state KV. The callback verifies the ID token's `iss` equals
   `expected_iss` (NOT the current `upstream_idp.issuer_url`, which an
   admin could have changed mid-flow).
2. **RFC 9207 `iss` callback param:** if the upstream supplies `iss` in
   the authorization response, validate it equals `expected_iss`.
   Absence is NOT an error (older IdPs don't supply it).
3. **PKCE S256:** every authorize URL includes `code_challenge` +
   `code_challenge_method=S256`. The `code_verifier` is stashed in
   state.
4. **Nonce:** every authorize URL includes `nonce`; the callback
   verifies the ID-token's `nonce` claim equals the stashed value.
5. **State single-use:** `kv.Store.Pop` — atomic consume.
6. **JWT alg allowlist:** RS256, ES256, EdDSA. Reject HS256, `none`,
   and anything else. Configured via `zitadel/oidc/v3` client options
   AND additionally verified post-hoc as defense-in-depth.
7. **Auth code single-use:** zitadel/oidc/v3's RP enforces this on the
   client side; the upstream OP enforces it server-side. We rely on
   both.

### D8. Discovery + JWKS caching

Use `zitadel/oidc/v3`'s built-in discovery doc cache and JWKS cache.

- Discovery doc: cached per `Cache-Control` headers from the IdP;
  fallback TTL 15 minutes.
- JWKS: cached with TTL ~15 minutes; refresh on key-not-found during
  verification (handles upstream key rotation transparently).

Implementer should NOT cache the discovery doc longer than 1 hour; an
IdP rotating their JWKS endpoint URL must propagate within reasonable
time.

### D9. Upstream error handling

Upstream OPs return errors via the callback URL:
`?error=access_denied&state=...&error_description=...`.

The callback handler:

1. Consume state (Pop) first. Validates the request belongs to a real
   flow. If state missing: 401 `state_invalid`.
2. If `error` parameter present: surface as HTTP 4xx with body
   `{"code":"upstream_error","upstream_code":"<error>","message":"<error_description or stable text>"}`.
3. If `error` absent and `code` present: proceed with code exchange.
4. If neither: 400 `invalid_callback`.

Audit emission for upstream errors: `factor:federation_oidc,
event:fail, detail:{reason:"upstream_error", upstream_code:"..."}`.

### D10. Username collision in auto_provision

Per master spec §"Threat model deltas": "if upstream's
`username_claim` value matches an existing local account with no link
to this IdP, reject (admin must intervene)."

**Implementation:** in `pkg/federation/oidc/modes.go:AutoProvision`,
the lookup sequence is:

1. `GetAccountIdentityByIssuerSub(upstream_iss, upstream_sub)` — if
   row exists: that's the local account; we're done.
2. If not found AND auto_provision permitted:
   - `GetAccountByUsername(upstream_username_claim_value)`.
   - If row found: there's a local account with this username that
     hasn't been linked to this IdP. Reject with 403, error code
     `username_collision`. Audit `factor:federation_oidc, event:fail,
     detail:{reason:"username_collision"}`.
   - If row not found: provision a new account with this username,
     display_name from upstream, role='user' (admin must be enrolled
     by other means), email_verified validation per D1, allowed_domains
     check.
3. After provisioning, insert `account_identity` row.

### D11. AAD for `upstream_idp.client_secret_enc`

Per master spec: AAD = `'upstream_idp:'+id+':'+key_version`. Same
construction pattern as TOTP secret AAD.

Implementation in `pkg/federation/oidc/secret.go`:

```go
func aadFor(idpID int64, keyVersion int32) []byte {
    return []byte("upstream_idp:" +
        strconv.FormatInt(idpID, 10) + ":" +
        strconv.Itoa(int(keyVersion)))
}
```

Use `crypto/aes` + `crypto/cipher.NewGCM` directly (same as
`pkg/credential/totp/aead.go`). Don't import a new crypto wrapper.

### D12. Disabled account check

The federation callback flow must re-fetch the account AFTER the
`account_identity` upsert and BEFORE issuing the session. If the
account is disabled at that moment: 403 `account_disabled`, no session
issued. Mirrors the v0.2 audit Bundle 1 step-2 disabled-check pattern.

### D13. Audit emission

All federation events write `credential_event` rows via
`audit.Writer.Record`:

- `factor:federation_oidc, event:register` — first-time auto_provision
  (`detail:{idp_slug:"...", upstream_iss:"...", reason:"auto_provision"}`).
- `factor:federation_oidc, event:link` — `/me/identities/link/.../callback`
  success (`detail:{idp_slug:"...", upstream_iss:"..."}`).
- `factor:federation_oidc, event:use` — every successful federation
  login (`detail:{idp_slug:"..."}`).
- `factor:federation_oidc, event:unlink` — `/me/identities/{id}/unlink`
  success (`detail:{idp_slug:"..."}`).
- `factor:federation_oidc, event:fail` — every callback failure
  (`detail:{reason:"...", idp_slug:"..."}`). Reasons include
  `email_not_verified`, `username_collision`, `state_invalid`,
  `iss_mismatch`, `nonce_mismatch`, `upstream_error`, `code_exchange_failed`,
  `id_token_invalid`, `account_disabled`, `invite_required`,
  `link_required`.

The `EventLink` and `EventUnlink` constants exist in
`pkg/audit/event.go` already.

### D14. No throttle on federation callbacks

The auth_throttle table (per-account, per-factor) is for credential
guessing. Federation callbacks don't accept user-submitted credentials
— the upstream OP did the credential check. Failed callbacks
(state-expired, mix-up rejection, etc.) bump an IP-level rate limit
only (the existing `s.rateLimit` helper with `"federation:ip:"` prefix,
30/min).

The login redirect (`/auth/federation/{slug}/login`) has the same IP
rate limit.

## Mock OP for smoke

Add an in-process mock OP in `cmd/smoke/mockop/` (or directly in
`cmd/smoke/main.go` if it stays under ~200 lines). Use
`zitadel/oidc/v3`'s test-OP helper if it exists; otherwise hand-roll a
minimal OP that serves:

- `GET /.well-known/openid-configuration` — discovery doc with
  `issuer`, `authorization_endpoint`, `token_endpoint`, `jwks_uri`,
  `userinfo_endpoint`, supported algs/scopes.
- `GET /authorize` — accepts the OAuth params, redirects to the
  Prohibitorum callback with `code` and `iss` params.
- `POST /token` — exchanges the code for an access + id token. Signs
  the ID token with a freshly-minted ES256 key.
- `GET /jwks` — publishes the signing key.

The mock OP exposes test hooks to inject:
- Specific claim values (email, email_verified, preferred_username,
  name, sub, iss, amr).
- Error responses on /authorize (for the upstream-error path).
- Wrong-issuer ID tokens (for the mix-up rejection test).

## Smoke extension

Append after the v0.2 step 45. New steps 46–60 (approximate):

```
46.  Insert upstream_idp row via direct SQL: slug='mockop', issuer_url=mock OP URL,
     client_id=test, client_secret_enc=<AES-GCM(...)>, mode='auto_provision',
     allowed_domains={'example.com'}, require_verified_email=true.
47.  GET /auth/federation/mockop/login → 302 to mock OP /authorize. Capture state cookie.
48.  Follow 302 to mock OP /authorize. Mock OP redirects with code+iss to /callback.
49.  GET /callback?code=...&state=...&iss=... → 302 to /, session cookie set.
50.  /me round-trip: confirm new account with username from upstream preferred_username,
     display_name from upstream name.
51.  DB assert: account_identity row exists with (upstream_iss=mockop_iss, upstream_sub=...).
     session row has upstream_idp_id set.
     credential_event has factor=federation_oidc, event=register and event=use rows.
52.  Logout.
53.  Repeat federation login with same upstream sub, but change upstream name claim →
     verify display_name updates locally on re-login.

54.  Negative: change mock OP to return email_verified=false. Hit /login → /callback.
     Expect 403 email_not_verified. DB assert: credential_event has detail.reason=email_not_verified.
55.  Restore email_verified=true.

56.  Username collision: register a local account with username "collide_user" via
     bootstrap enrollment. Have mock OP return preferred_username=collide_user.
     /login → /callback for that user. Expect 403 username_collision.

57.  link_only mode: insert a second upstream_idp row with mode='link_only'.
     Attempt /auth/federation/<slug>/login as a new user (no prior link) → expect 403
     link_required.

58.  Self-service link: log in as smoke-admin via WebAuthn. POST /me/identities/link/<slug>/begin
     → 302 to mock OP. Complete flow → 204. DB assert: account_identity row for smoke-admin.

59.  /me/identities → 200 with the linked IdP row.

60.  Unlink last sign-in method (negative): create an account with ONLY a federated identity
     (no WebAuthn, no password). Attempt /me/identities/{id}/unlink. Expect 400
     last_sign_in_method.

61.  Unlink with backup: that same account adds a passkey via /me/credentials/register.
     Now unlink the federated identity. Expect 204.
```

The mock OP runs alongside the dev server on a different port (e.g.,
:8090). cmd/smoke starts both before running steps.

## Verification gates (per `feedback_always_verify_fixes.md`)

v0.3 is not done until:

1. `go build ./...` clean.
2. `go test ./...` passes (unit tests for federation, modes, secret,
   handlers).
3. `cmd/smoke` runs steps 1–~61 green against:
   - The dev server (built from this branch).
   - A live Postgres.
   - The in-process mock OP.
4. All smoke DB-state assertions pass (account_identity rows,
   credential_event rows for federation_oidc factor).
5. STATUS.md, AUDIT.md, INTEGRATION.md updated to reflect v0.3 surface
   and smoke coverage. Honest "Smoke-untested runtime paths" list per
   v0.1.1 / v0.2 discipline.

## Post-implementation audit

Following the v0.2 pattern: after the implementation lands and smoke
passes, dispatch a parallel audit (crypto + race/logic + standards)
against the v0.3 commit range. Address findings before declaring
production-ready.

Specific federation-relevant standards to verify against:
- RFC 9700 (OAuth 2.0 Security BCP)
- RFC 9207 (`iss` callback parameter)
- OIDC Core 1.0 §2, §3 (ID token validation)
- OIDC Discovery 1.0
- RFC 8414 (OAuth metadata)
- RFC 7636 (PKCE)
- RFC 8176 (`amr` values)

  pass-through-or-empty default is honest.
