# v0.6 Protocol Completeness — design

**Goal:** Close the deferred OIDC OP + SAML IdP behaviors tracked across the
v0.4/v0.5 audits, staying backend (no frontend), so the IdP interoperates with
mainstream OIDC RPs and SAML SPs. Four work areas: forced re-authentication
(cross-protocol), OIDC PKCE-method policy + introspection auth, SAML
`NameIDPolicy/@Format` honoring, and SAML IdP-initiated SSO. Every decision is
grounded in how production IdPs/SPs actually behave (see the Research appendix);
where the spec and mainstream behavior diverge, we follow what is least likely
to break real-world interop while staying secure.

**Architecture:** Extend the existing `pkg/protocol/oidc` (Provider) and
`pkg/protocol/saml` (IdP) handlers in place, following the established v0.4/v0.5
patterns (Approach B — hand-rolled handlers over the DB-backed schema + the
session/KV/audit infrastructure). No new packages. The same `signing_key` RSA
key continues to sign both protocols.

**Prior art / context:** `docs/superpowers/notes/2026-05-30-session-handoff.md`
(v0.5 end state + the deferred list), `AUDIT.md` (the deferred rows being
closed), the v0.4 spec `docs/superpowers/specs/2026-05-29-v0.4-oidc-op-downstream-design.md`
and v0.5 spec `docs/superpowers/specs/2026-05-30-v0.5-saml-idp-design.md`.

## Decisions

- **D1 — Forced re-auth = full fresh re-login + per-request freshness gate.**
  `prompt=login` / `max_age`-exceeded (OIDC) and `ForceAuthn=true` (SAML) cause
  the IdP to refuse to issue from the current session and bounce to the login
  flow; the user re-authenticates via the normal ceremony (which mints a session
  with a fresh `auth_time`); on return the handler confirms the re-auth happened
  *for this request* and issues. This matches Shibboleth / FusionAuth / Clerk /
  Keycloak (Research §1) and OIDC Core §3.1.2.1 ("the OP SHOULD prompt the
  End-User for re-authentication"). The lighter in-place step-up was rejected as
  a looser reading.

- **D2 — Freshness gate via a single-use KV marker keyed by a nonce.** When the
  IdP demands re-auth it stamps `{oidc,saml}:reauth:<nonce> = <demand_instant>`
  (TTL 10m), embeds `<nonce>` in the `return_to` it bounces to, and on return
  requires `session.auth_time >= demand_instant` AND the marker to still exist
  (then consumes it, single-use). A stale pre-existing session cannot satisfy
  `prompt=login`/`ForceAuthn` because its `auth_time` predates the demand. This
  is the concrete realization of the standard "`auth_time` prevents stale-session
  satisfaction" property (Research §1). The two protocols share one
  `reauthGate` helper (distinct key prefixes).

- **D3 — `max_age` semantics.** `max_age=N` → demand re-auth iff
  `now - session.auth_time > N` (with the ±clock-skew already used elsewhere);
  `max_age=0` ≡ always demand (== `prompt=login`); absent → no demand. The
  issued `id_token.auth_time` reflects the (possibly refreshed) value, which RP
  libraries validate against `max_age` (Research §1).

- **D4 — `prompt=none` + re-auth demand → `login_required`** (no bounce), the
  existing pattern. `prompt` may be a space-delimited set; `login` and `none`
  together is invalid → `invalid_request`.

- **D5 — SAML `ForceAuthn` + `IsPassive` both true → `NoPassive`.** IsPassive
  wins (OASIS SAML Core normative; Research §1). `ForceAuthn` + an existing
  session (non-passive) → the D1/D2 re-auth bounce. After re-auth, the assertion
  `AuthnInstant` reflects the fresh `auth_time`.

- **D6 — OIDC PKCE method policy: S256-only by default, `plain` configurable but
  off.** `/oauth/authorize` consults the existing `oidc_client.require_pkce`
  (if true, `code_challenge` mandatory) and `allowed_code_challenge_methods`
  (the requested method must be in the set). A DB CHECK / policy keeps `plain`
  out of the default set; an operator may add it per-client but it is never the
  default. S256-only is universal and safe in 2025–26 (RFC 9700, OAuth 2.1;
  Research §2).

- **D7 — Token introspection requires an authenticated (confidential) client;
  public clients may NOT introspect.** RFC 7662 §2.1 requires authorization for
  every introspection caller (callers are resource servers) — there is no
  public-client exemption (Research §2). `/oauth/introspect` from a `none`-auth
  (public) client → `invalid_client` (401). This is a **behavior change** to the
  v0.4 endpoint, which previously allowed public clients to introspect their own
  tokens. Revocation (RFC 7009) is unchanged: public clients may still revoke
  their **own** tokens (ownership-checked), which RFC 7009 explicitly permits.

- **D8 — SAML `NameIDPolicy/@Format`: honor-if-producible, `unspecified`/absent
  is free, `Requester`/`InvalidNameIDPolicy` on genuine mismatch.** If the
  AuthnRequest carries `NameIDPolicy/@Format` that is present, non-empty, not
  `urn:oasis:names:tc:SAML:2.0:nameid-format:unspecified`, and not equal to the
  format we can produce (`sp.NameIDFormat`, the persistent-1.1 default), the IdP
  returns a `<Response>` with top-level StatusCode `Responder` nested
  `InvalidNameIDPolicy` and NO assertion. Absent/`unspecified` → proceed with the
  SP's configured format. This matches Shibboleth/ADFS/Entra (Research §3) and
  SAMLCore §3.4.1.1 (per the OASIS errata). (Top-level status is `Responder`
  with the `InvalidNameIDPolicy` second-level code, per SAMLCore — the practical
  recommendation phrased it "Requester/InvalidNameIDPolicy"; we use the
  spec-literal `Responder` top + `InvalidNameIDPolicy` sub. Confirm the exact
  pairing against SAMLCore §3.2.2.2 at implementation; both are second-level
  under a top-level status and SPs key on the second-level `InvalidNameIDPolicy`.)

- **D9 — SAML POST-binding AuthnRequest intake.** `HandleSSO` accepts the
  HTTP-POST binding (form `SAMLRequest`, base64 with NO inflate, **enveloped**
  signature verified via `verifyElementSignature`) in addition to the existing
  HTTP-Redirect binding. The `POST /saml/sso` route is re-added and the POST SSO
  binding is re-advertised in IdP metadata (reversing the v0.5 audit-batch-D
  removal, now that the intake is real).

- **D10 — Signed IdP metadata + freshness.** The `EntityDescriptor` is signed
  (given an `ID`, `signElement` with the active key) and carries `validUntil`
  (now + a configurable window) and `cacheDuration`. Optional Metadata fields,
  but they let SPs cache-bound + integrity-check (Research §4 ecosystem norm).

- **D11 — SAML IdP-initiated SSO with per-SP opt-in.** A new
  `GET /saml/sso/init?sp=<entity_id>[&RelayState=…]` app-launcher endpoint emits
  an **unsolicited** Response (no `InResponseTo`; `SubjectConfirmationData`
  without `InResponseTo`) to the SP's **default ACS**. Gated behind a new
  per-SP `saml_sp.allow_idp_initiated` boolean (**default false**) — the IdP
  refuses (`403`/direct error) for SPs that did not opt in. This mirrors GHES's
  own posture (GHES rejects unsolicited Responses by default; Research §4).
  `RelayState` is passed through **verbatim** as the SP's deep-link/target (the
  Okta/AWS convention). The assertion goes only to the registered default ACS
  (open-redirect guard) and relies on the existing short validity window +
  SessionIndex + AudienceRestriction as replay mitigations; the inherently
  weaker login-CSRF posture (no `InResponseTo`, SAML Profiles §4.1.5) is
  documented.

- **D12 — No new auth UI.** The login bounce targets the existing
  `/login?return_to=…` (operator/dashboard-provided, unchanged); the actual
  re-authentication is the existing API ceremony. v0.6 adds no frontend. The
  smoke drives re-auth via the API ceremony then retries the protocol request,
  exactly as v0.4/v0.5 smoke drives the no-session case.

## Data flow — forced re-auth (shared shape)

1. Protocol request arrives (`/oauth/authorize` or `/saml/sso`), session present.
2. Compute the re-auth demand: OIDC `prompt`/`max_age` vs `session.auth_time`;
   SAML `ForceAuthn`. No demand → proceed (existing flow).
3. Demand + `prompt=none`/passive → `login_required`/`NoPassive` (no bounce).
4. Demand (interactive): mint nonce; `SetEx("<proto>:reauth:<nonce>", now, 10m)`;
   redirect to `/login?return_to=<original request URL + &reauth=<nonce>>`.
5. User re-authenticates (API ceremony → `Issue()` mints session, `auth_time=now`).
6. Browser returns to the original request URL (now carrying `reauth=<nonce>`).
7. Handler reads the nonce, looks up the marker `T`; requires
   `session.auth_time >= T` and marker present; consumes the marker. Satisfied →
   issue (with the fresh `auth_time`/`AuthnInstant`). Not satisfied (missing/expired
   marker, or `auth_time < T`) → re-demand (back to step 4) — a stale session
   cannot pass.

A single `reauthGate` helper encapsulates steps 2/4/7 for both protocols.

## Error handling

- OIDC: `invalid_request` (prompt=login+none), `login_required` (prompt=none +
  demand), `invalid_client` (public-client introspection, D7), the existing
  `invalid_request`/`invalid_scope` PKCE errors (D6). All on the spec's
  error-channel side (direct vs redirect) per the v0.4 open-redirect ordering.
- SAML: `NoPassive` status Response (ForceAuthn+IsPassive, D5), `Responder`/
  `InvalidNameIDPolicy` status Response (D8), direct `403`/error for a
  non-opted-in IdP-initiated SP (D11) and for an unknown SP (existing). No
  redirect to an untrusted target on any parse/validation failure (existing
  guard preserved).

## Testing

- Unit tests per handler change (the `reauthGate` helper gets focused tests:
  stale session does NOT satisfy `prompt=login`; nonce is single-use; expired
  marker re-demands; `max_age` boundary).
- `cmd/smoke` extended (steps continuing from 99): `prompt=login`
  bounce-then-reauth (assert it does NOT issue from the stale session, then
  issues after a fresh API login with a newer `auth_time`); `max_age` stale→fresh;
  SAML `ForceAuthn` bounce-then-reauth; `ForceAuthn`+`IsPassive` → NoPassive;
  PKCE method rejection (a non-allowed method → error); public-client introspect
  → `invalid_client`; `NameIDPolicy` mismatch → `InvalidNameIDPolicy`;
  POST-binding AuthnRequest round-trip; signed-metadata signature verify;
  IdP-initiated SSO accepted by the mock SP (with `allow_idp_initiated=true`) and
  refused when the flag is false.
- After all tasks land: the same parallel post-implementation audit
  (crypto/XML-DSig + protocol-standards + race/logic + deep integration/data/
  schema-drift), special focus on: the re-auth freshness gate (can a stale
  session satisfy `prompt=login`? can the nonce be replayed across flows or
  users? does the demand→return survive a concurrent login?), the introspection
  auth change (no public-client bypass), `NameIDPolicy` status correctness, and
  IdP-initiated open-redirect (must POST only to the registered default ACS, only
  for opted-in SPs).

## Research appendix — mainstream-compatibility findings (2026-05-31)

Sourced via a multi-angle web search pass over authoritative + vendor docs
(OIDC Core, OAuth 2.1 / RFC 9700, RFC 7636/7662/7009, OASIS SAML Core/Profiles,
OWASP, and Okta/Auth0/Entra/Keycloak/Shibboleth/GHES/Salesforce/AWS docs).
The deep-research harness's adversarial-verification stage did not complete, so
these are search-phase syntheses cross-checked against the named specs; the
load-bearing claims (RFC 7662 caller-auth, OASIS IsPassive-wins, GHES
unsolicited-default-reject, S256-universal) are well-established and were
confirmed against the primary specs during design.

1. **Forced re-auth.** OIDC `prompt=login` → full re-authentication; `auth_time`
   is the enforcement mechanism (the returned `auth_time` must post-date the
   request; RP libraries validate it against `max_age`). `max_age` MAY be
   satisfied by a per-factor step-up (Curity) but full re-login is the safe
   superset (Shibboleth/FusionAuth/Clerk). SAML: IsPassive wins over ForceAuthn
   when both set → `NoPassive` (OASIS normative). Keycloak maps
   `prompt=login`⇄`ForceAuthn`. → **D1–D5.**
2. **PKCE + introspection.** S256-only is universal/safe in 2025–26 (RFC 9700,
   OAuth 2.1); rejecting `plain` by default is safe (Authlete/WorkOS). RFC 7662
   requires auth for **all** introspection callers (no public-client exemption);
   RFC 7009 permits public clients to revoke their **own** tokens. → **D6, D7.**
3. **NameIDPolicy/@Format.** Real SPs do send Format (AWS Cognito `persistent`,
   GHES configurable, ADFS `unspecified`). Spec is strict (SAMLCore §3.4.1.1):
   honor a requested non-`unspecified` Format or return an error; `unspecified`
   frees the IdP. Standards-strict IdPs (Shibboleth/ADFS/Entra) honor-or-error;
   the interop-safe posture is honor-if-producible + `unspecified`-is-free +
   `InvalidNameIDPolicy` on genuine mismatch. → **D8.**
4. **IdP-initiated SSO.** GHES (our primary target) **rejects unsolicited
   Responses by default** and bounces back an AuthnRequest, behind an explicit
   opt-in toggle. OWASP frames IdP-initiated as login-CSRF/replay-weaker (no
   InResponseTo, SAML Profiles §4.1.5). Mainstream IdPs expose it via an
   SP-identifying URL; `RelayState` carries the deep-link/target (AWS/Okta).
   Mitigations: short validity window, SessionIndex/one-time-use, audience
   restriction, deliver only to the registered default ACS. → **D10 (metadata),
   D11 (per-SP opt-in + guardrails).**
